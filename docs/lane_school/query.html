En detaylı sayfaKod objesi → tensor → lane karşılığı

Query tam olarak nedir? İçinde ne var? Lane’e nasıl dönüşüyor?

Bu sayfa “query karttır” gibi kısa benzetmeleri bırakıp, kodun bellekte ne tuttuğunu ve bunun görüntüde hangi çizgiye dönüştüğünü tek tek anlatır.

En önemli düzeltme: query içinde x1, x2, x3 koordinatları yok. Query başlangıçta 256 sayılık öğrenilen bir vektördür. Koordinatlar daha sonra head/decoder tarafından üretilir.

1. En Başta Elimizde Ne Var?

Modelin nihai amacı bu görseldeki şeritleri nokta nokta üretmek. Ama bunu doğrudan “resimden x1,x2,x3 çıkar” diye yapmıyor. Arada birkaç temsil var.

Image tensor

images yaklaşık [B, 3, 288, 800]. Bu gerçek RGB görüntü.

Feature map

features yaklaşık [B, 256, 72, 200]. Bu artık piksel değil; görüntünün sıkıştırılmış anlam haritası.

Lane output

pred_x_rows [B, 20, 72]. İşte gerçek çizgi noktaları burada doğuyor.

2. Query Bir Array mi?

Evet, query PyTorch tarafında bir tensör/array. Ama koordinat array’i değil. Kod şunu kuruyor:

# dynlaneseq_eg/modeling/lane_queries.py
self.query = nn.Embedding(num_slots=20, dim=256)

# self.query.weight shape:
[20, 256]

Yani modelin içinde 20 tane satır var. Her satır 256 float sayıdan oluşuyor.

query.weight =
slot 0: [0.013, -0.004, 0.021, ..., 0.006]  # 256 sayı
slot 1: [-0.009, 0.017, 0.003, ..., -0.014] # 256 sayı
...
slot 19: [0.002, 0.011, -0.018, ..., 0.020]
Bu 256 sayının tek tek “x”, “y”, “renk”, “sol lane” diye insan tarafından okunabilir anlamı yok. Model eğitim sırasında bu sayıları ayarlıyor. Bu vektör, “ben bir lane adayı olarak görüntüye nasıl bakmalıyım?” bilgisini taşımaya başlıyor.

Slot kutusu çizginin kendisi değildir. Slot kutusu çizgiyi arayacak başlangıç vektörüdür.

3. Query Batch İçinde Nasıl Çoğalıyor?

Bir batch içinde B tane görüntü varsa aynı 20 query her görüntü için kopyalanır:

# lane_queries.py
q = self.query.weight.unsqueeze(0)      # [1, 20, 256]
q = q.expand(batch_size, -1, -1)        # [B, 20, 256]

Bu ne demek?

Batch’teki her görüntüye 20 aday lane arayıcısı verilir.

q[b, n, :] = b’inci görüntüde n’inci lane adayının 256 sayılık vektörü.

Bu ne demek değil?

q[b, n, 0] x koordinatı değildir.

q[b, n, 1] y koordinatı değildir.

Bu sayılar direkt çizilemez. Çizgiye dönüşmeleri için head gerekir.

4. Query Görüntüye Nasıl Bakıyor?

Backbone ve FPN görüntüyü feature map’e çevirir. Sonra bu feature map düzleştirilir:

# dynlaneseq_s0.py
f_proj = self.proj(fpn)                         # [B, 256, 72, 200]
mem_value = f_proj.flatten(2).transpose(1, 2)   # [B, 14400, 256]
mem_key = (f_proj + pos).flatten(2).transpose(1, 2)

14400 = 72 * 200. Yani feature map’te 14.400 küçük konum var. Her konumun 256 sayılık anlam vektörü var.

Cross-attention şu işi yapar: query vektörü, 14.400 feature konumu içinden kendisi için önemli bilgiyi toplar.

# cross_attention_decoder.py
q_content = cross_attn(
    query=q_with_pos,     # lane adayı
    key=memory_key,       # görüntü konumları + pozisyon
    value=memory_value    # görüntü konumlarının içerikleri
)
Cross-attention sonrası q1 hâlâ [B,20,256]. Shape değişmez. Ama artık query vektörü görüntüden bilgi almıştır. Başta boş arayıcıydı; şimdi “bu görüntüde muhtemelen şöyle bir lane var” bilgisini içinde taşır.

5. Peki Query Ne Zaman Çizgi Oluyor?

S0’da çizgi, query’den geçen üç head ile doğar:

# heads_s0.py
exist_logits = self.exist(q)       # [B, 20, 2]
row_x_logits = self.row_x(q)       # [B, 20, 72, 200]
pred_x_rows = soft_expected_x(row_x_logits) # [B, 20, 72]
range_norm = self.range(q)         # [B, 20, 2]

exist_logits

Bu slot gerçekten lane mi, boş mu?

Kodda class 0 = lane, class 1 = no-lane gibi kullanılıyor. Çünkü matcher p_lane = softmax(...)[...,0] alıyor.

range_norm

Lane dikeyde nereden başlıyor, nerede bitiyor?

Bu x değil. İki sayı: y_min ve y_max. Sonradan 288 piksele ölçeklenir.

6. row_x_logits İçinde Ne Var?

Bu, çizgiye en yakın tensor. Shape:

row_x_logits[b, n, p, k]

b = batch içindeki görüntü
n = lane slotu, 0..19
p = y satırı, 0..71
k = x kutusu, 0..199

Görüntü genişliği 800 px. x_bins = 200. Bir kutu yaklaşık 4 piksel eder.

k = 0   -> x ≈ 0 px
k = 50  -> x ≈ 200 px
k = 100 -> x ≈ 400 px
k = 150 -> x ≈ 600 px
k = 199 -> x ≈ 796 px

soft_expected_x bu 200 kutuyu sert seçmez. Olasılık dağılımının ağırlıklı ortalamasını alır:

probs = softmax(row_x_logits)
expected_bin = sum(probs * bin_index)
x_pixel = expected_bin * (800 / 200)

Sonuç: pred_x_rows[b,n,p]. Bu artık çizilebilir x koordinatıdır.

7. Lane Çizgisi Tam Olarak Hangi Tensörden Çiziliyor?

Final çizgi çoğu yerde pred_x_rows ile çizilir. y koordinatları ayrıca sabittir:

# common.py
y_rows = [0, 4, 8, 12, ..., 284]  # 72 değer

# bir lane çizgisi:
points = [
  (pred_x_rows[b,n,0],   0),
  (pred_x_rows[b,n,1],   4),
  (pred_x_rows[b,n,2],   8),
  ...
  (pred_x_rows[b,n,71], 284),
]

8. S1’de Query’ye Ne Ekleniyor?

S1’de query tek başına 72 row’u bir MLP ile üretmek yerine, her row için ayrı token yapılır.

# dynlaneseq_s1.py
row_emb = self.row_embedding.weight.view(1, 1, 72, 256)
tokens = queries.unsqueeze(2) + row_emb

# shape:
queries.unsqueeze(2) = [B, 20, 1, 256]
row_emb              = [1, 1, 72, 256]
tokens               = [B, 20, 72, 256]

Bu şu demek: aynı lane adayı 72 farklı y satırına kopyalanıyor. Her kopyaya “ben şu y satırıyım” bilgisi ekleniyor.

tokens[b,n,p,:]

b’inci görüntü, n’inci lane, p’inci y satırı için 256 sayılık temsil.

Lane karşılığı

Bu, çizgi üstündeki tek boncuğun bellekteki halidir. Henüz x değil. X, row decoder’dan sonra çıkar.

9. S2’de Evidence Tam Olarak Nedir?

S2’de önce coarse_x üretilir. Bu kaba çizginin x koordinatlarıdır:

coarse_x = coarse_row["pred_x_rows"]  # [B, 20, 72]

Sonra sampler bu x noktalarının altındaki feature’ları okur:

sample_x = curriculum.build_sample_x(coarse_x, ...)
evidence = sampler(features, sample_x)

# evidence shape:
[B, 20, 72, 256]

evidence[b,n,p,:] artık şu anlama gelir: “b’inci görüntüde, n’inci lane adayının p’inci row noktasında görüntüden okunan 256 sayılık kanıt”.

S1’de tokens[b,n,p,:] sadece query + row bilgisiydi. S2’de buna gerçek görüntüden o noktada okunan evidence[b,n,p,:] eklenir. İşte fark bu.

10. S2’de Kod Lane’i Nasıl Değiştiriyor?

# dynlaneseq_s2.py
tokens = queries.unsqueeze(2) + row_emb + self.adapter(evidence)
row = self.row_decoder(tokens)
final_x = row["pred_x_rows"]

Bu satırın görsel anlamı:

queries

“Ben hangi lane adayının boncuğuyum?”

row_emb

“Ben bu lane’in hangi y satırındayım?”

adapter(evidence)

“Bu noktada görüntü bana ne söylüyor?”

Final decoder bu üç bilgiyi birleştirip yeşil çizgiyi üretir.

11. S3’te Bridge Hangi Array’i Değiştiriyor?

S3 query’yi doğrudan çizgiye çevirmiyor. S2’den gelen evidence array’ini değiştiriyor.

# before bridge
evidence: [B, 20, 72, 256]

# low_rank_bridge.py
out[bi, ni] = e + bridge_scale * delta

# after bridge
refined_evidence: [B, 20, 72, 256]

Yani shape aynı kalıyor. İçerik değişiyor. Lane görselinde bu, sarı noktalardan gelen ham kanıtın final çizgiye daha uygun hale getirilmesi gibi düşünülebilir.

12. S4’te İkinci Bakış Hangi Tensor ile Oluyor?

S4’te stage1 final çizgisi, stage2 için yeni sample çizgisi olur.

# dynlaneseq_s4.py
stage1_x = stage1_out["final"]["pred_x_rows"]  # [B,20,72]
sample_x2 = curriculum.build_sample_x(stage1_x, ...)
evidence2 = sampler(features, sample_x2)       # [B,20,72,256]
stage2 = row_decoder(query + row + evidence2 + hidden_scale*h1)

  gerçek kamera görüntüsü

  [B,256,72,200]

memory = flatten(features)

  görüntüden bilgi almış lane adayları

row_x_logits
  [B,20,72,200]
  her lane, her row için 200 x kutusu skoru

  [B,20,2]

  [B,20,72,256]
  çizgi üstünden okunan görüntü kanıtı

stage2.pred_x_rows  # S4

  ikinci bakıştan sonra final çizgi

14. En Önemli Yanlış Anlamaları Temizleyelim

Query koordinat değildir

Query başlangıçta [20,256] learned embedding tablosudur. İçinde açık x1,x2,x3 yoktur.

Koordinat pred_x_rows ile doğar

Çizilebilir lane noktaları row_x_logits -> soft_expected_x sonrası oluşur.

Evidence görüntüden okunmuş feature’dır

Bu, lane üstündeki noktalarda feature map’ten alınan 256 sayılık vektördür.

Bridge çizgiyi doğrudan çizmez

S3 bridge evidence içeriğini değiştirir; çizgi yine decoder sonrası çıkar.

Bu Sayfayı Tek Cümlede Hatırla

Query, koordinat listesi değil; görüntüye bakıp lane çizgisine dönüşmek için kullanılan 256 sayılık aday temsilidir. Çizgi ise pred_x_rows tensoründe doğar.
